크림 리셀 수익 모니터링 봇 — 프로젝트 설명서
이 프로젝트가 뭔지
한마디로 말하면, 패션 쇼핑몰에서 싸게 사서 크림(KREAM)에서 비싸게 파는 걸 자동으로 찾아주는 시스템이다.

크림은 한국의 한정판/패션 리셀 플랫폼인데, 여기서 신발이나 옷이 정가보다 비싸게 거래되는 경우가 많다. 반대로 무신사, 나이키, 카시나 같은 일반 쇼핑몰에서는 회원 등급 할인, 쿠폰, 세일 등으로 같은 상품을 정가보다 훨씬 싸게 살 수 있다.

핵심은 이거다. 같은 상품인데 A 사이트에서는 8만원에 살 수 있고, 크림에서는 13만원에 팔 수 있다면 수수료 빼고도 돈이 남는다. 이런 기회를 사람이 일일이 찾는 건 불가능하니까, 봇이 24시간 자동으로 모니터링하면서 수익이 나는 조합을 발견하면 디스코드로 알려주는 거다.

중요한 건 이게 선착순 구매 경쟁이 아니라는 점이다. 누구보다 빨리 사자가 아니라, 회원 등급 할인이나 쿠폰으로 남들보다 싸게 살 수 있는 가격을 기준으로 차익을 계산하는 정보 기반 시스템이다. 봇은 "이거 사면 얼마 남는다"라고 알려주기만 하고, 실제 구매 판단과 실행은 사용자가 직접 한다.

누가 쓰는지
이 시스템은 1인 운영용이다. 사용자는 크림 판매 이력이 수천 건인 숙련된 리셀러이고, 사업자 택배를 사용하기 때문에 배송비가 일반인보다 저렴하다. 크림의 수수료 구조, 검수 과정, 시세 흐름을 이미 잘 알고 있어서 봇이 정보를 주면 빠르게 판단할 수 있는 사람이다.

다루는 상품 범위
시작은 신발(스니커즈)로 한다. 이후 의류, 모자, 악세서리 등 패션 카테고리로 확장할 예정이고, 전자기기 같은 건 당분간 제외한다. 사이트도 마찬가지로 처음엔 무신사와 크림으로 시작하고, 나이키, 카시나, 그 외 사이트는 하나씩 추가해 나간다. 그래서 설계 자체가 새로운 사이트나 카테고리를 쉽게 붙일 수 있는 모듈 구조여야 한다.

시스템이 하는 일을 순서대로 설명하면
첫 번째, 쇼핑몰 크롤링이다. 무신사 같은 리테일 사이트에 로그인한 상태로 접속해서 상품을 검색한다. 로그인이 핵심인 이유는 회원 등급에 따라 할인가가 다르기 때문이다. 비로그인 상태에서 보이는 가격이랑 실제 내 계정으로 보는 가격이 다를 수 있다. 여기서 상품명, 모델번호, 할인 적용된 실제 구매가, 사이즈별 재고 여부를 수집한다.

두 번째, 크림에서 같은 상품을 찾는다. 리테일 사이트에서 가져온 모델번호를 크림에서 검색해서 정확히 같은 상품을 찾는다. 여기서 "유사 상품"이 아니라 모델번호가 정확히 일치하는 같은 상품만 매칭해야 한다. 유사 상품끼리 비교하면 의미가 없으니까.

세 번째, 크림에서 상세 시세를 가져온다. 단순히 "이 신발이 대충 12만원"이 아니라, 사이즈별로 즉시판매가가 얼마인지, 최근에 실제로 거래가 됐는지, 일주일에 몇 건이나 거래되는지, 사려는 사람(구매 입찰)이 몇 명인지를 수집한다. 같은 신발이라도 사이즈 260은 13만원에 팔리는데 270은 8만원일 수 있고, 어떤 사이즈는 한 달에 한 번밖에 안 팔릴 수 있기 때문이다.

네 번째, 사이즈별로 수익을 정밀 계산한다. 크림 판매 수수료(6% + 부가세), 검수비(2,500원), 크림 배송비(3,500원), 사업자 택배비를 전부 빼고 실제로 손에 쥐는 순수익을 계산한다. 투자 대비 수익률(ROI)도 함께 보여준다.

다섯 번째, 조건에 맞으면 디스코드로 알린다. 순수익이 일정 금액 이상이고 거래량도 충분하면 알림을 보낸다. 알림에는 상품 정보, 구매처별 가격(같은 상품이 여러 사이트에 있으면 전부 보여줌), 사이즈별 크림 시세와 순수익, 거래 활성도, 수수료 상세 내역, 구매 링크가 한눈에 들어와야 한다. 이미 알린 상품의 가격이 변동되면(추가 할인이나 크림 시세 변화) 가격 변동 알림도 따로 보낸다.

여섯 번째, 이 과정을 30분마다 자동 반복한다. 기본 주기는 30분이지만, 수익 상품이 감지되면 해당 상품만 더 짧은 간격(10분)으로 추적한다. 매일 자정에는 하루 요약 리포트도 자동 생성한다.

기술적으로 어떻게 만드는지
개발 언어는 Python 3.13이고, 디스코드 봇은 discord.py 라이브러리를 사용한다. 가격 이력이나 알림 기록은 SQLite 로컬 데이터베이스에 저장한다.

크롤링이 이 프로젝트에서 가장 까다로운 부분이다. 크림(KREAM)은 일반적인 Playwright 직접 접속을 차단한다. 테스트 결과 Page.goto에서 net::ERR_HTTP_RESPONSE_CODE_FAILURE 오류가 발생하며, 이는 서버 측에서 자동화 브라우저를 감지하고 거부하는 것이다. 이 문제를 해결하기 위해 Chrome CDP(Chrome DevTools Protocol) 방식을 사용한다. 실제 크롬 브라우저를 원격 디버깅 모드로 하나 띄워놓고, Playwright가 그 브라우저에 연결해서 조종하는 방식이다. 사람이 브라우저를 쓰는 것과 동일하게 동작하기 때문에 차단되지 않는다. 이 방식은 실제 테스트에서 성공적으로 작동이 확인되었다.

크롬 인스턴스는 반드시 하나만 유지하고 탭을 재사용해야 한다. 상품마다 새 크롬을 열고 닫으면 프로세스가 수백 개 생기고 시스템 리소스를 잡아먹는 문제가 실제로 발생했다. 크롬 창은 --window-position=-2400,-2400 옵션으로 화면 밖에 숨기거나, 가능하다면 --headless=new 모드로 실행한다. 단, 크림이 headless를 차단할 경우 화면 밖 방식으로 대체한다.

크림은 로그인이 필수다. 비로그인 상태에서는 체결 내역이 5건으로 제한되고, 사이즈별 즉시구매가/즉시판매가 페이지(/buy/select/, /sell/select/)에 접근할 수 없다. 로그인 방식은 자동 로그인이 아니라 수동 1회 로그인이다. 처음 한 번만 CDP 크롬에서 직접 로그인하면 쿠키가 chrome-debug 디렉토리에 저장되어 이후 자동으로 로그인 상태가 유지된다. 이 방식이 자동 로그인보다 차단 위험이 낮고 안정적이다.

무신사도 등급 할인가를 가져오려면 반드시 로그인 상태에서 크롤링해야 한다. 무신사는 크림과 달리 Playwright 직접 접속이 가능하므로, Playwright로 로그인 세션을 유지하면서 크롤링하거나, 크림과 같은 CDP 방식을 사용할 수 있다. 어떤 방식이 더 안정적인지는 개발 과정에서 테스트해서 결정한다.

개발 환경은 Windows PC + WSL(Ubuntu)이다. 클로드 코드를 사용해서 개발하며, 클로드 코드는 WSL 위에서 실행된다. 단, Chrome은 Windows 측에 설치되어 있으므로 CDP 연결 시 Windows의 Chrome을 사용하는 점을 고려해야 한다. 프로젝트 파일은 Windows 바탕화면의 크림봇 폴더(/mnt/c/Users/USER/Desktop/크림봇)에 위치한다. 안정화 이후에는 AWS, GCP, 또는 네이버 클라우드 서버로 이전할 수 있으며, 사용자가 서버 운영 경험이 있기 때문에 이 부분은 문제없다.

과도한 요청으로 차단당하지 않도록 요청 사이에 랜덤 딜레이(3~8초)를 넣는다. 에러가 발생하면 자동 재시도하되, 크롬이 완전히 죽으면 자동 재시작한다. 기존에 사용 중인 일반 크롬 브라우저를 종료하지 않도록, 디버그 크롬만 식별해서 관리해야 한다.

봇과 소통하는 방법
디스코드 서버에 채널을 용도별로 나눠서 운영한다. 수익 알림 채널에는 수익 기회가 발견되면 자동으로 알림이 올라온다. 가격 변동 채널에는 이미 감지된 상품의 가격이 바뀌었을 때 알림이 온다. 일일 리포트 채널에는 매일 자정에 하루 요약이 자동 생성된다. 수동 검색 채널에서는 사용자가 명령어를 입력하고 결과를 받는다. 설정 채널에서는 수수료율이나 수익 기준 같은 봇 설정을 변경한다. 로그 채널에는 시스템 상태와 에러가 기록된다.

사용할 수 있는 명령어는 즉시 스캔(!스캔), 크림 상품 상세 조회(!크림 상품ID), 모델번호로 전 사이트 비교(!비교 모델번호), 무신사 검색(!무신사 키워드), 모니터링 키워드 관리(!키워드, !키워드추가, !키워드삭제), 설정 조회 및 변경(!설정), 수동 리포트(!리포트), 봇 상태 확인(!상태), 도움말(!도움) 등이 있다.

수수료 계산 방식
크림에서 상품을 팔 때 드는 비용은 이렇다. 판매 수수료는 판매가의 6%에 부가세 10%를 곱한 금액이고, 검수비 2,500원과 크림 배송비 3,500원이 고정으로 든다. 구매처(무신사 등)에서 오는 배송비는 대부분 무료이고, 크림으로 보내는 택배비는 사업자 택배 단가를 적용한다. 순수익은 크림 즉시판매가에서 구매가와 이 모든 비용을 뺀 금액이다. 수익률(ROI)은 순수익을 구매가로 나눈 퍼센트로 함께 표시한다.

알림에는 순수익 금액(원)과 수익률(%) 둘 다 표시한다. 수수료, 검수비, 배송비 등 비용 내역도 상세히 보여줘서 사용자가 계산을 직접 검증할 수 있게 한다.

시그널 기준은 순수익 3만원 이상이고 7일 거래량이 10건 이상이면 강력매수, 순수익 1만5천원 이상이고 거래량 5건 이상이면 매수, 순수익 5천원 이상이면 관망, 그 이하이거나 거래량이 3건 미만이면 비추천이다. 이 기준은 모두 설정에서 자유롭게 변경할 수 있다.

알림에 꼭 들어가야 하는 정보
수익 기회 알림 하나에는 다음이 전부 포함되어야 한다. 상품명과 모델번호, 모든 구매처의 가격(같은 상품이 무신사와 카시나에 둘 다 있으면 둘 다 보여줌, 물량 확보를 위해), 각 구매처의 할인 정보(정가 대비 할인율, 할인 유형), 사이즈별 크림 즉시판매가와 순수익(원)과 수익률(%), 사이즈별 구매 입찰 수(사려는 사람이 몇 명인지), 사이즈별 구매처 재고 여부(그 사이즈를 실제로 살 수 있는지), 7일 거래량과 30일 거래량과 마지막 거래일과 가격 추세, 수수료 상세 내역(크림 수수료, 검수비, 배송비 각각), 구매 링크, 감지 시각.

가격 변동 알림에는 어떤 가격이 얼마에서 얼마로 바뀌었는지, 순수익이 얼마나 변했는지, 시그널이 바뀌었으면 바뀐 시그널을 보여준다.

일일 리포트에는 하루 동안의 스캔 횟수, 비교한 상품 수, 발견한 수익 기회 수, 순수익 기준 상위 5개 상품 요약을 포함한다.

개발 순서
1단계는 기반 구축이다. 프로젝트 폴더 구조를 잡고, 크림 CDP 연결과 로그인 세션을 설정하고, 크림 사이즈별 시세 크롤러를 만들고, 수익 계산 엔진과 데이터베이스를 만든다.

2단계는 수집 레이어다. 무신사 로그인 크롤러를 만들어서 등급 할인가와 사이즈 재고를 가져오고, 모델번호 기반 정확 매칭을 구현한다. 가격 변동 감지 로직도 여기서 만든다.

3단계는 비교와 알림이다. 사이즈별 정밀 비교 엔진, 시그널 판정, 디스코드 알림 메시지 생성, 명령어 핸들러를 완성한다.

4단계는 자동화와 안정화다. 주기적 자동 스캔, 수익 상품 집중 추적, 가격 변동 알림, 일일 리포트, 중복 알림 방지, 에러 처리와 크롬 자동 복구를 구현한다.

5단계는 확장이다. 나이키, 카시나 등 추가 사이트를 모듈로 붙이고, 의류와 악세서리 카테고리를 추가하고, 서버 배포를 진행한다.

이 프로젝트의 성공 기준
모델번호 매칭 정확도가 95% 이상이어야 한다. 크림 상품과 리테일 상품이 진짜 같은 제품인지 정확히 판별하는 것이 가장 중요하다. 사이즈별 가격 수집 성공률은 90% 이상, 가격 반영 지연은 30분 이내, 수익 계산 오차는 500원 이내, 시스템 가동률은 95% 이상이 목표다.
